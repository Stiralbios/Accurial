# Schemas and Types

The frontend uses three sources of types and schemas. Each has a specific role.

## The Three Sources

| Source | Generated by | Used for |
|---|---|---|
| **Generated TS types** | Orval (from OpenAPI) | Request/response types, never edited |
| **Zod schemas** (handwritten) | You | Form input validation |
| **TS interfaces / type aliases** (handwritten) | You | UI-only shapes that don't exist on the backend |

## Generated Types — the source of truth for API DTOs

Orval emits TypeScript interfaces matching every Pydantic schema:

```ts
// app/services/api/generated/model/questionRead.ts
export interface QuestionRead {
  id: string;
  title: string;
  description: string;
  prediction_type: PredictionType;
  status: QuestionStatus;
  owner_id: string;
}
```

Re-export from the generated client at the call site:

```ts
import type { QuestionRead, QuestionCreate } from "@/services/api/generated";
```

## Zod Schemas — for form validation

Forms need runtime validation, not just compile-time types. Each feature with a form defines Zod schemas in `features/<feature>/schemas.ts`:

```ts
// app/features/question/schemas.ts
import { z } from "zod";

export const questionCreateSchema = z.object({
  title: z.string().min(1).max(255),
  description: z.string().min(1),
  prediction_type: z.enum(["BINARY"]),
});

export type QuestionCreateInput = z.infer<typeof questionCreateSchema>;
```

`QuestionCreateInput` is used as the form's input type and **must structurally match** the `QuestionCreate` generated type. ESLint cannot enforce the equivalence; we use a TS-level check:

```ts
import type { QuestionCreate } from "@/services/api/generated";

// Compile-time assertion that the Zod-derived type matches the generated DTO.
type _AssertQuestionCreate = QuestionCreateInput extends QuestionCreate ? true : never;
type _AssertReverse = QuestionCreate extends QuestionCreateInput ? true : never;
const _check: _AssertQuestionCreate & _AssertReverse = true;
```

This pattern catches drift between handwritten Zod schemas and the OpenAPI-generated types at compile time.

> **Future**: a generator like `zod-from-openapi` may replace these handwritten schemas. Until adopted, the manual mirror with the assertion is the convention.

## Naming Convention

Mirrors the backend's schema strategy.

| Variant | Backend (Pydantic) | Frontend Zod | Frontend TS type |
|---|---|---|---|
| Read | `QuestionRead` | (not needed — generated) | `QuestionRead` (generated) |
| Create | `QuestionCreate` | `questionCreateSchema` | `QuestionCreateInput` |
| Update | `QuestionUpdate` | `questionUpdateSchema` | `QuestionUpdateInput` |
| Filter | `QuestionFilter` | `questionFilterSchema` | `QuestionFilterInput` |

Internal-only schemas (`<Feature>Internal`, `<Feature>CreateInternal`) have **no frontend equivalent** — they are backend-only.

## Internal vs External — Frontend Edition

Even though the frontend doesn't have the same API↔Service↔Store split, an analogous boundary exists between **form input** and **API payload**.

```
Form input  (RHF state, validated by Zod)
    |
    | submit handler maps form -> API
    v
API payload  (generated Create / Update type)
    |
    v
Mutation hook
```

Most of the time these are the same shape. When they diverge (e.g., form has a `confirmPassword` field, API doesn't), the form schema is a **superset** that maps down before submission:

```ts
const passwordFormSchema = z.object({
  password: z.string().min(8),
  confirmPassword: z.string(),
}).refine((d) => d.password === d.confirmPassword, { path: ["confirmPassword"] });

type PasswordFormInput = z.infer<typeof passwordFormSchema>;

function toApiPayload(form: PasswordFormInput): UserCreate {
  return { password: form.password, /* ... */ };
}
```

## Enums

Backend `StrEnum` values become string-literal unions in the generated client:

```ts
export type QuestionStatus = "DRAFT" | "OPEN" | "CLOSED" | "ARCHIVED";
```

For UI labels, define a separate constant — never repurpose the API value as a label:

```ts
// app/features/question/constants.ts
import type { QuestionStatus } from "@/services/api/generated";

export const QUESTION_STATUS_LABEL: Record<QuestionStatus, string> = {
  DRAFT: "Draft",
  OPEN: "Open",
  CLOSED: "Closed",
  ARCHIVED: "Archived",
};
```

## UUIDs and Dates

- IDs are `string` (UUID v7 from the backend). Treat as opaque.
- Dates from the backend are ISO-8601 strings. Convert to `Date` at the rendering layer using a helper from `app/lib/date.ts`. Never store `Date` objects in TanStack Query cache — they don't serialize well.

## Don't

- Don't redeclare types that exist in the generated client.
- Don't widen generated types (e.g., adding `| undefined` where the API guarantees a value).
- Don't import generated types into `app/components/` shared components — those should be domain-agnostic. Pass primitives or cross-feature types instead.
