# Frontend Architecture Overview

High-level design of the Accurial frontend.

## System Diagram

```
+------------------+
|   Browser        |
|   React 19 SPA   |
+--------+---------+
         |
         | HTTP (httpOnly cookie auth, JSON)
         |  credentials: 'include'
         v
+--------+---------+
| FastAPI Backend  |
+------------------+
```

## Layered Architecture

The frontend follows a layered architecture symmetric to the backend's API → Service → Store split. UI code never talks to the network directly.

```
Component (React)
    |
    | calls
    v
Feature hook  (use<Feature><Action>)
    |
    | wraps
    v
Generated TanStack Query hook  (Orval)
    |
    | uses
    v
fetch wrapper  (credentials, error normalization)
    |
    v
Backend HTTP API
```

| Layer | Location | Responsibility |
|---|---|---|
| Routes | `app/routes/` | TanStack Router file-based routes; thin shells that compose components |
| Components | `app/features/<feature>/components/` and `app/components/` | Render UI, no data fetching of their own beyond hooks |
| Feature hooks | `app/features/<feature>/hooks/` | Wrap generated query/mutation hooks with feature semantics, expose the surface needed by UI |
| Generated client | `app/services/api/generated/` | Auto-produced by Orval from `openapi.json`; never edited by hand |
| Fetch wrapper | `app/services/api/client.ts` | Sets `credentials: 'include'`, base URL, normalizes `BaseProblem` errors |
| Stores (Zustand) | `app/stores/` and `app/features/<feature>/stores/` | UI/client state only — auth status, modals, filters, drafts |
| Query client | `app/services/query-client.ts` | TanStack `QueryClient` singleton, defaults |

## Server vs Client State

Frontend state is split along a strict boundary inspired by [TkDodo's "Practical React Query"](https://tkdodo.eu/blog/practical-react-query):

- **Server state** (anything that originates from the backend) — owned by **TanStack Query**. Never mirror server data into a Zustand store.
- **Client state** (UI toggles, current user identity flag, transient filter values, optimistic drafts before submit) — owned by **Zustand**.

See `state-management.md` for rules and examples.

## Request Lifecycle

1. A route loader or component mounts.
2. A feature hook is called. It dispatches a generated TanStack Query hook (`useQuery` or `useMutation`).
3. The generated hook calls the typed Orval function which calls the fetch wrapper.
4. The fetch wrapper sends the request with `credentials: 'include'` so the browser attaches the auth cookie.
5. The backend validates the cookie via `get_current_user`, runs the API → Service → Store flow, returns JSON.
6. On success: data is normalized via the response Zod-or-OpenAPI types and cached by TanStack Query under a stable query key.
7. On failure: the wrapper parses the `BaseProblem` envelope (`{type, on, title, detail, status}`) and throws an `ApiError`.
8. UI consumes either `data` or shows the appropriate error state. See `error-handling.md`.

## Authentication Flow (target)

The target authentication transport is the **httpOnly cookie** set by the backend. The full flow lives in `features/auth.md` along with the backend prerequisites.

## Cross-cutting Concerns

| Concern | Solution |
|---|---|
| Routing | TanStack Router (file-based) with route-level loaders that prefetch via TanStack Query |
| Forms | React Hook Form + Zod resolver |
| Styling | Tailwind CSS, design tokens in `tailwind.config.ts` |
| Errors | `ApiError` class + global error boundary + per-form inline mapping |
| Tests | Vitest + RTL + MSW with handlers generated by Orval |

See per-topic documents in `conventions/` for details.
